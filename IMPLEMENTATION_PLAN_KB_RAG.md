# План реализации Knowledge Base + RAG для SupportBot

## Цель

Добавить автоматические ответы клиентам в Telegram на основе базы знаний, не ломая текущий поток работы менеджеров и сохранив простую поддержку решения в будущем.

Основные требования:
- авто-ответ только на текстовые сообщения;
- лимит авто-ответов на клиента в сутки;
- fallback на менеджера, если релевантного ответа нет;
- управление базой знаний из админки;
- отображение ИИ-ответов в общем чате менеджеров.

---

## 1. Принципы реализации

Чтобы решение было поддерживаемым, лучше держаться таких правил:

- бизнес-правила в одном месте, чтобы не дублировать лимиты и пороги;
- векторный поиск и генерацию ответов отделить от UI;
- хранить KB-данные отдельно от embeddings;
- все AI-ответы логировать как обычные сообщения в системе;
- предусмотреть расширение без миграции всей архитектуры.

---

## 2. Рекомендуемая архитектура

## Контур

```text
Telegram -> Supabase webhook function -> message saved
                                     -> pre-checks
                                     -> AI reply service
                                     -> Telegram reply
                                     -> save ai_bot message

Admin UI -> KB CRUD -> KB tables
                     -> embedding refresh
```

## Где должна жить AI-логика

Рекомендуемый вариант:
- webhook в `supabase/functions/telegram-webhook/` остаётся точкой входа для Telegram;
- RAG-логика живёт в `support-admin/src/lib/rag/`;
- webhook вызывает внутренний HTTP endpoint в `support-admin`, например `/api/ai/auto-reply`;
- endpoint возвращает структурированный результат, а webhook уже решает, отправлять ли ответ клиенту.

Почему так:
- OpenAI и RAG-слой проще поддерживать рядом с основным backend-кодом Next.js;
- проще тестировать и расширять API;
- UI, KB CRUD и AI-логика остаются в одном приложении.

Обязательное условие:
- нужен внутренний `INTERNAL_APP_URL` или аналогичный env для вызова Next.js API из Supabase function;
- endpoint должен быть защищён внутренним токеном, а не быть публичным.

Если не хочется зависеть от HTTP между Supabase function и Next.js:
- можно потом перенести RAG целиком в Supabase Edge Function;
- но на первом этапе логика в `support-admin` обычно удобнее в разработке.

## 3. Реальная структура проекта

В проекте фронт и API находятся не в корне, а в:
- `support-admin/src/...`

Поэтому все новые серверные модули и API лучше добавлять туда:
- `support-admin/src/lib/openai.ts`
- `support-admin/src/lib/rag/...`
- `support-admin/src/app/api/ai/auto-reply/route.ts`

Telegram webhook остаётся в:
- `supabase/functions/telegram-webhook/...`

## 4. Функциональные правила

## До вызова RAG

Проверки должны выполняться в таком порядке:

1. сообщение текстовое;
2. длина текста допустима;
3. не нарушен rate-limit;
4. сообщение не является простым приветствием;
5. не превышен дневной лимит;
6. только после этого вызывается RAG.

## Рекомендуемые правила

- Тип сообщения: только текст.
- Длина сообщения: до 500 символов.
- Rate-limit: не чаще 1 авто-ответа на клиента в 30 секунд.
- Дневной лимит: 20 успешных авто-ответов на клиента в сутки.
- Greeting-ответ: считается авто-ответом и входит и в rate-limit, и в дневной лимит.
- Fallback: не входит в дневной лимит.

Это убирает двусмысленность и делает поведение предсказуемым.

---

## 5. Что считать confidence

Самое важное уточнение: confidence нужно привязать не к “ощущению модели”, а к понятному правилу.

Рекомендуемый первый вариант:
- confidence = similarity лучшего найденного KB-результата;
- если лучший результат ниже порога, генерация не делается;
- если лучший результат выше порога, в prompt передаются top-3 результата.

Рекомендуемый стартовый порог:
- начать с `0.75`;
- после реальных тестов подстроить до `0.70-0.80`.

Причина:
- similarity проще анализировать;
- это стабильнее и прозрачнее, чем просить LLM “самооценить уверенность”.

- отображается как отдельный тип отправителя;
- участвует в истории переписки как обычное сообщение.

## 7. Поиск и генерация

## Рекомендуемая структура модулей

```text
support-admin/src/lib/rag/
  types.ts
  embed.ts
  search.ts
  generate.ts
  pipeline.ts
```

## Ответственность модулей

- `embed.ts`:
  создаёт embedding из пользовательского запроса;

- `search.ts`:
  ищет релевантные KB-записи через Supabase RPC или SQL;

- `generate.ts`:
  формирует ответ только по найденному контексту;

- `pipeline.ts`:
  объединяет этапы и возвращает единый результат:
  `shouldReply`, `answer`, `confidence`, `matchedEntryIds`, `reason`.

## Правила генерации

Системный prompt должен быть простым:
- отвечай только на основе переданного контекста;
- не выдумывай факты;
- если контекста недостаточно, верни отказ от ответа;
- отвечай кратко и по-русски;
- не обещай действий, которых система реально не делает.

---

## 8. Векторизация KB

Для первой версии лучше делать так:

- при создании статьи сохранять KB-запись;
- затем запускать embedding;
- если embedding не построился, статья остаётся в статусе `embedding_failed`;
- менеджер может вручную перезапустить векторизацию.

Это лучше, чем делать вид, что статья доступна, хотя в поиске её ещё нет.

Рекомендуемое правило пересчёта:
- если изменились `question`, `answer` или `keywords`, пересчитать embedding;
- если changed hash совпадает, не пересчитывать.

---

## 9. Интеграция в Telegram webhook

## Текущее направление

Webhook уже принимает сообщение и сохраняет его. Это правильно, менять общий входной поток не нужно.

Новый порядок работы:

1. сохранить входящее сообщение клиента;
2. выполнить pre-checks;
3. если нужен greeting, отправить greeting;
4. если лимит или ограничения нарушены, отправить fallback для менеджера или промолчать;
5. если проверки пройдены, вызвать AI endpoint;
6. если `shouldReply = true`, отправить ответ;
7. сохранить ответ как `ai_bot`;
8. увеличить дневной usage;
9. записать событие в AI audit-таблицу.

## Важно

Лимит и окончательное решение лучше держать в одном слое.

Рекомендуемый вариант:
- webhook отвечает за transport и pre-checks;
- endpoint `/api/ai/auto-reply` отвечает только за RAG decision;
- учёт лимита и rate-limit выполняется в webhook.

Так проще избежать двойной логики.

## 10. API слой

## AI endpoint

`support-admin/src/app/api/ai/auto-reply/route.ts`

Принимает:
- `messageText`
- `clientId`
- `sourceMessageId`

Возвращает:
- `shouldReply`
- `answer`
- `confidence`
- `matchedEntryIds`
- `reason`

Примеры `reason`:
- `match_found`
- `low_confidence`
- `no_match`
- `generation_refused`

## KB API

Набор endpoints:
- список статей;
- создание статьи;
- редактирование статьи;
- удаление статьи;
- список категорий;
- создание и редактирование категорий;
- ручной пересчёт embedding.

Важно:
- CRUD API должно быть только для менеджеров;
- embedding-trigger лучше запускать на сервере, а не на клиенте.

## 11. Admin UI

## Цели UI

Минимальный необходимый интерфейс:
- список категорий;
- список статей;
- поиск по названиям/вопросам;
- форма редактирования;
- статус векторизации;
- кнопка “пересчитать embedding”.

## Интеграция

Поскольку админка уже существует, KB лучше встроить как отдельный режим панели:
- переключение между чатами и базой знаний;
- без отдельного приложения;
- с повторным использованием текущих layout-паттернов.

## Что важно для поддержки

- не смешивать UI формы и серверные вызовы в одном большом компоненте;
- список, форма и управление категориями разделить на отдельные компоненты;
- хранить типы KB отдельно от chat-типов.

---

## 12. Отображение AI-сообщений в чате

AI-ответы должны быть видны менеджерам так же, как ручные ответы.

Нужно:
- добавить `ai_bot` в типы;
- показать отдельную подпись, например `ИИ Ассистент`;
- задать визуальное отличие;
- не ломать текущую логику unread и сортировки.

Особое внимание:
- в коде сейчас много мест, где логика завязана на “если не manager, значит client”;
- при добавлении `ai_bot` такие места нужно пройти внимательно.

Это несложная задача, но она затрагивает несколько компонентов, а не только один тип.

## 13. Порядок реализации

Рекомендуемый порядок:

1. Миграции БД:
   KB таблицы, embeddings, usage, `sender_type = ai_bot`.
2. Серверные типы и OpenAI клиент.
3. RAG модули в `support-admin/src/lib/rag/`.
4. AI endpoint `/api/ai/auto-reply`.
5. Доработка Supabase webhook.
6. CRUD API для KB.
7. KB UI в админке.
8. Отображение `ai_bot` в сообщениях.
9. Тестирование на реальных сценариях.

Такой порядок минимизирует переделки.

## 14. Что считать первой рабочей версией

MVP можно считать готовым, когда:

- менеджер может добавить статью в KB;
- у статьи строится embedding;
- клиент пишет текстовый вопрос в Telegram;
- при хорошем совпадении бот отвечает автоматически;
- при плохом совпадении отправляется fallback;
- при достижении лимита бот больше не отвечает автоматически;
- менеджер видит AI-ответ в чате.

## 15. Расширение в будущем

Чтобы решение было удобно развивать, стоит уже сейчас оставить место для таких расширений:

- несколько embeddings на одну статью через чанки;
- ручное подтверждение AI-ответа менеджером;
- многоязычная KB;
- аналитика: top вопросов без ответа;
- обучение на реальных вопросах клиентов;
- переиндексация KB по расписанию;
- A/B тест порогов similarity;
- разные политики для разных категорий вопросов;
- отключение AI по расписанию или по статусу команды.

---

## 16. Риски

Главные риски:

- дублирование логики лимитов между webhook и API;
- слишком оптимистичный confidence threshold;
- некорректная обработка `ai_bot` в существующем UI;
- синхронная векторизация может замедлить сохранение статьи;
- отсутствие audit-логов затруднит отладку.

---

## 17. Итоговая рекомендация

Реализовывать это стоит, но в чуть более строгом виде, чем в исходном плане:

- использовать реальные пути проекта `support-admin/src/...`;
- явно разделить transport, business rules и RAG;
- не дублировать лимиты в нескольких слоях;
- считать confidence через similarity поиска;
- сразу заложить status для KB и базовый audit AI-событий;
- добавлять `ai_bot` как полноценный тип сообщения, а не как исключение.

Такой вариант остаётся компактным, но будет заметно проще в поддержке и расширении.
